Setup

One-line install — clones the repo, installs Rust if needed, builds, and generates an access token into .env:

curl -fsSL https://raw.githubusercontent.com/I-SHOW-AKIRU200/picodb/main/install.sh | bash

Or clone and drive the interactive CLI manually:

git clone https://github.com/I-SHOW-AKIRU200/picodb.git && cd picodb && ./picodb

The CLI is fully scriptable too:

./picodb setup     # install Rust (if needed), build, generate token
./picodb run       # start the server (loads .env)
./picodb test      # run the integration tests
./picodb token     # regenerate the access token
./picodb status    # build / auth / config status

Once running, open the dashboard at this address and paste the token from .env when prompted. Two ports are exposed:

PortPurpose
7120Raw binary wire protocol — the database engine.
7121HTTP dashboard, /metrics, /api/keys, and WebSocket /ws.

Connection URI

Clients connect to the engine port (7120) using a picodb:// URI:

picodb://[username:password@]host[:port]     # default port 7120

The part before @ is your auth secret. It must match the server's PICODB_TOKEN, or its PICODB_USERNAME/PICODB_PASSWORD. With a single shared token, leave the username empty:

picodb://:<token>@HOST:7120
The picodb:// URI is a client-side convention the example clients parse (host + port + AUTH). The server itself speaks the raw binary protocol — it doesn't parse URIs.

Client libraries

Zero-dependency reference clients ship in the repo: examples/client.py and examples/client.js.

Python

from client import PicoDB
db = PicoDB.from_uri("picodb://:<token>@localhost:7120")
db.set("user:1", "alice", ttl=3600)
print(db.get("user:1"))          # b'alice'
db.delete("user:1")

Node.js

const { PicoDB } = require("./client");
const db = await PicoDB.connect("picodb://:<token>@localhost:7120");
await db.set("user:1", "alice", 3600);
console.log((await db.get("user:1")).toString());  // "alice"
db.close();

Data types

Beyond plain strings, PicoDB supports Redis-style hashes, lists, and sets. A key holds one type; the wrong command on a key returns a WRONGTYPE error, and collections that become empty are deleted automatically.

TypeCommands
stringSET (TTL) · GET · DEL
hashHSET · HGET · HDEL · HGETALL · HLEN
listLPUSH · RPUSH · LPOP · RPOP · LRANGE · LLEN
setSADD · SREM · SMEMBERS · SISMEMBER · SCARD
anyTYPE · DEL · FLUSH
# hashes
db.hset("user:1", "email", "a@x.com");  db.hgetall("user:1")
# lists
db.rpush("queue", "a", "b");  db.lrange("queue", 0, -1)
# sets
db.sadd("tags", "x", "y", "x");  db.scard("tags")      # 2 (deduped)
db.type("user:1")   # 'hash' | 'list' | 'set' | 'string' | 'none'

Authentication

One shared secret guards every surface. Set it in .env two interchangeable ways:

Style.envConnect with
Single tokenPICODB_TOKEN=<token>picodb://:<token>@host:7120
User + passwordPICODB_USERNAME=admin
PICODB_PASSWORD=secret		picodb://admin:secret@host:7120

How each surface authenticates:

  • Binary engine (7120) — send the AUTH command with the token before other commands (the clients do this for you).
  • HTTP API (7121) — send Authorization: Bearer <token> (for curl / Prometheus).
  • Browser dashboard — enter the token once; it is exchanged at POST /login for an HttpOnly, SameSite=Strict session cookie. The token is never placed in a URL.
Regenerate a fresh random token anytime with ./picodb token.

Dashboard & HTTP API

Everything on port 7121:

RouteWhat it returns
GET /The live dashboard (open; exposes no cache data).
GET /docsThis page.
POST /loginExchange a Bearer token for the session cookie.
POST /logoutClear the session cookie.
GET /metricsPrometheus exposition auth
GET /api/keysJSON snapshot of all keys + stats auth
GET /wsWebSocket: live stats push + pub/sub bridge auth

Scrape metrics with a Bearer token

curl -H "Authorization: Bearer <token>" http://HOST:7121/metrics

For Prometheus, set authorization: { type: Bearer, credentials: <token> } on the scrape job.

Configuration

All runtime settings live in .env (copy from .env.example) — no code changes needed.

Env varDefaultMeaning
PICODB_BIND127.0.0.1Bind address. 0.0.0.0 exposes on all interfaces (public).
PICODB_ENGINE_PORT7120TCP port for the raw binary engine.
PICODB_HTTP_PORT7121TCP port for the dashboard / metrics / API / WebSocket.
PICODB_TOKENunsetShared access token. Set → auth enforced everywhere. Unset → auth disabled (loopback dev).
PICODB_MAX_BYTES52428800Hard RAM cap (50 MiB); oldest keys evicted (LRU) past it.

Security

  • Token auth on the engine, HTTP API, and WebSocket, with a constant-time comparison; the token is never logged.
  • No token in URLs — the dashboard uses an HttpOnly + SameSite=Strict cookie, so the secret can't leak via logs, history, or Referer, and JS/XSS can't read it.
  • No built-in TLS. Traffic is plaintext on the wire — for untrusted networks, terminate TLS at a reverse proxy (nginx / caddy) in front of :7121 and :7120.
Exposing publicly (PICODB_BIND=0.0.0.0)? Always set PICODB_TOKEN, open the port in your firewall, and put TLS in front — otherwise the token travels in cleartext.
